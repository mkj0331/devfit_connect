# 1️⃣ GitHub 레포 요약 & 분석 리포트 생성

### 🎯 AI가 해야 할 일

사용자의 GitHub 레포를 보고

- **무슨 기술을 썼는지**
- **어떤 프로젝트인지**
- **어느 수준인지**
- **강점이 뭔지**

→ **사람이 읽은 것처럼 정리된 리포트 생성**

### 🔹 입력 데이터 (AI에게 줄 것)

- README.md
- requirements.txt / package.json
- 폴더 구조
- 주요 코드 파일 (전부 ❌, 핵심만 ⭕)
- 커밋 메시지 요약

## 추천 모델

- `gpt-4.1` / `gpt-4.1-mini`

### 왜 모델 학습 안 하고 API 쓰는가?

- “코드를 읽고 의미를 해석”하는 건
    
    👉 **학습 난이도 + 비용 + 데이터 부족** 때문에 현실적으로 힘듦
    
- LLM은 이미:
    - 코드 이해
    - 프로젝트 요약
    - 기술 수준 평가를 다 할 수 있음
- 사용 방식 예시
    
    ```python
    [시스템 프롬프트]
    너는 개발자 채용을 평가하는 AI다.
    
    [유저 입력]
    - README.md
    - 폴더 구조
    - 핵심 코드 요약
    
    [출력]
    - 프로젝트 개요
    - 사용 기술 스택
    - 구현 난이도
    - 채용 시장 기준 강점
    ```
    

### 🔹 (보조) 경량 모델로 가능한 것

| 역할 | 방법 |
| --- | --- |
| 언어 감지 | regex / rule |
| 기술 키워드 추출 | TF-IDF / KeyBERT |
| 파일 중요도 판단 | rule 기반 |

👉 **하지만 “사람처럼 해석”은 불가능**

# 3️⃣ 자소서 문항 초안 생성

### 🎯 AI가 해야 할 일

- 사용자 경험
- 채용공고 요구사항
- 기업 정보

👉 **이 셋을 엮어서 문항별 초안 작성**

### 🔹 입력 데이터 묶기

```json
{
"user":"...",
"job_posting":"...",
"company_analysis":"...",
"question":"지원 동기와 입사 후 포부를 작성하시오."
}
```

---

## 🔹 여기서 LLM은 필수

이건:

- 논리 구성
- 문맥 연결
- 말투 조절

→ **규칙 / 모델로 절대 불가능**

## 🔹 왜 Model + Fine-Tuning 안 하는가?

| 이유 | 설명 |
| --- | --- |
| 데이터 부족 | 자소서 정답 데이터 없음 |
| 기업별 상이 | 일반화 어려움 |
| 프롬프트가 더 유연 | 즉시 수정 가능 |

- OpenAI API를 너무 많이 쓰는거 아니야??
    
    # 1️⃣ 먼저 결론부터 (아주 명확하게)
    
    > ❌ 이 프로젝트는 “모델 안 써서 아쉬운 프로젝트”가 아님
    > 
    > 
    > ✅ 오히려 **“모델을 안 쓴 게 올바른 AI 설계 판단”인 프로젝트**
    > 
    
    **이걸 말로 설명할 수 있느냐가 실력 차이**야.
    
    ---
    
    # 2️⃣ 이 프로젝트에서 “모델 파인튜닝”이 부적합한 이유
    
    ## 🔹 ① 데이터가 구조적으로 없음
    
    ### 파인튜닝에 필요한 것
    
    ```
    입력 (X) : GitHub 레포 + 공고 + 기업 정보
    정답 (Y) : 이 사람에게 적합한 요약 / 추천 / 자소서
    ```
    
    ❗ **이런 정답 데이터는 세상에 존재하지 않음**
    
    - 자소서 = 정답이 없는 생성 문제
    - 추천 = 개인별 취향 + 맥락 문제
    - 레포 평가 = 주관적 판단
    
    👉 **Fine-Tuning 자체가 성립 안 됨**
    
    ---
    
    ## 🔹 ② 문제 자체가 “범용 언어·추론 능력” 문제
    
    이 프로젝트 AI가 하는 일은:
    
    - 코드 해석
    - 문맥 요약
    - 요구사항 비교
    - 글쓰기
    
    이건:
    
    > ❌ “도메인 특화 분류 문제”가 아님
    > 
    > 
    > ✅ “고차원 언어 추론 문제”
    > 
    
    👉 이미 **GPT급 모델만 가능한 영역**
    
    ---
    
    ## 🔹 ③ 파인튜닝 = 성능 향상 ❌ / 제약 증가 ⭕
    
    파인튜닝하면:
    
    - 말투 고정
    - 유연성 감소
    - 기업별/공고별 대응력 하락
    
    👉 **이 프로젝트랑 정반대**

    | --- | --- | --- |

    | 공고 요약 | ✅ 필수 | 문맥 요약 |
    | 자소서 생성 | ✅ 필수 | 생성형 추론 |

    ### ⚠️ 단, 전부 API만 쓰는 건 아님 (여기서 차별화됨)

    ---
    
    # 4️⃣ “API 썼네?”를 “AI 설계 잘했네”로 바꾸는 포인트
    
    ## 🔥 차별화 포인트 ①
    
    ### LLM은 “판단자”, 모델은 “필터”
    
    | 역할 | 방식 |
    | --- | --- |
    | 대량 공고 필터링 | 임베딩 + 벡터 유사도 |
    | 상위 후보 | LLM 정밀 판단 |
    
    👉 **비용·속도·정확도 균형 설계**
    
    ---
    
    ## 🔥 차별화 포인트 ②
    
    ### Prompt Engineering = 모델 설계
    
    이건 진짜야.
    
    - 역할 분리 프롬프트
    - 출력 구조 강제
    - 과장 방지 규칙
    - 자소서 문항별 템플릿
    
    👉 **이게 없으면 결과물 50% 수준**
    
    ---
    
    ## 🔥 차별화 포인트 ③
    
    ### AI 파이프라인 설계
    
    > AI = 모델이 아니라 흐름
    > 
    
    ```
    데이터 수집
    → 전처리
    → 임베딩
    → 후보 필터링
    → LLM 판단
    → 사용자 맞춤 생성
    ```
    
    이 구조를 **의도적으로 설계**했다는 게 핵심
    
    ---
    
    > 대규모 언어 모델을 직접 학습하는 대신,
    > 
    > 문제 특성상 범용 추론 능력이 요구된다고 판단하여
    > 
    > 사전 학습된 LLM을 추론 엔진으로 활용했습니다.
    > 
    > 대신, 임베딩 기반 추천, 규칙 기반 필터링,
    > 
    > 단계별 프롬프트 설계를 통해
    > 
    > 비용·정확도·확장성을 고려한 AI 파이프라인를 설계했습니다.
